## ADR-004: 顧客向けアカウント機能の深さ（Phase 1）
- **ステータス:** 仮決定
- **背景:** Phase 1 で顧客ログイン（予約履歴確認等）まで作るか、ゲスト予約のみにするか
- **決定:** Phase 1 はゲスト予約のみ。顧客アカウントは Phase 2 以降でLINEログインと共に導入。
- **理由:** 予約のハードルを最小化するため。会員登録を求めると離脱率が上がる。

## 確定済みの判断

### 技術スタック
- **決定:** React + TypeScript + Vite + Supabase + Tailwind CSS
- **理由:** 開発者（颯馬）の技術スタックに合致。Supabase は RLS でマルチテナント対応が容易。将来のSaaS展開にも対応可能。

### マルチテナント設計
- **決定:** company → store → staff の3階層 + RLS
- **理由:** 現在は1社3店舗だが、将来の他社導入を見据えてマルチテナント設計にしておく。

### Phase 1 スコープ
- **決定:** LP + Web予約 + 管理画面（最小構成）
- **理由:** ウェブ集客が最優先。予約の受け皿なしにLP集客しても意味がない。表裏一体でセット構築。
